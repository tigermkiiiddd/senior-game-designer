# 游戏开发规划

## 何时使用

- 启动新游戏项目或向发行商推介
- 确定开发资源和功能优先级
- 定义设计目标和项目范围
- 做出战略性项目管理决策
- 向团队或利益相关者传达项目方向

## 规划工作流

### 步骤 1：定义游戏支柱

游戏支柱是指导所有开发决策的高级、动作导向的概念。

**要求：**
- 定义**3 个支柱**（6 个太多了）
- 每个支柱**必须**与玩家**动作**相关，而不仅仅是美术/主题
- 组合成**一句话**用于沟通

**头脑风暴问题：**
- 如果我们结合两个类型的最佳元素会怎样？
- 如果我们修改另一个游戏中失败的机制会怎样？
- 如果我们扩展当前游戏玩法以允许新行为会怎样？
- 如果我们复制其他媒体的体验会怎样？

**示例——"流畅移动"作为支柱：**
- 摄像机：第三人称（突出移动优势）
- 角色设计：高瘦比矮壮更合适
- 战斗：必须将流畅移动延伸到战斗
- 世界设计：建筑必须传达可攀爬性
- 动画：需要许多过渡（计划工作量）

### 步骤 2：创建问题陈述

问题陈述是创作过程中**第一个**也是**最有影响力**的步骤。

**三个必需组成部分：**

1. **具体焦点** - 确定范围：整个游戏？某个系统？特定机制？
2. **可量化结果** - 必须产生可操作数据
3. **清晰沟通** - 通过向另一位设计师解释来测试

**范围校准：**

| 级别 | 示例 | 问题 |
|-------|---------|------|
| 太宽 | "如何设计回合制策略游戏？" | 浪费时间，不够详细 |
| 合适 | "如何创建带有轻度 RPG 元素和武器锻造的动作游戏？" | 界限清晰，可衡量目标 |
| 太窄 | "如何构建高级敌人巡逻 AI？" | 可能排除非常规解决方案 |

**验证方法：** 向熟悉问题和方法的合作设计师解释。

### 步骤 3：应用 80/20 法则进行资源分配

玩家花费约 80% 的时间使用约 20% 的游戏功能。

**战略行动：**
- 识别 20% 核心功能（基本动作：跳跃、战斗、升级）
- 集中开发精力完善这些核心功能
- 避免过度优化花哨但很少使用的功能

**应用示例：**
- **塞尔达**：玩家花费 80% 时间移动/攻击 - 优先平滑核心机制
- **魔兽移动**：地图穿越频繁 - 优化移动速度以防止沮丧
- **魔兽终局**：玩家在终局 80% 努力用于突袭（20% 内容）
- **游戏开始**：开始比结尾重玩更多 - 相应投资

### 步骤 4：管理项目约束

使用快-好-便宜三角做出明确的权衡。

**约束三角：**

```
        快
        /\
       /  \
      /    \
     /      \
   好 ----- 便宜
```

**你只能选择两个：**

| 选择 | 结果 | 策略 |
|--------|--------|----------|
| 快 + 便宜 | 不够好 | 原型制作、可行性检查 |
| 快 + 好 | 不便宜 | 雇用更多员工/专家 |
| 便宜 + 好 | 不快 | 独立开发、长时间线 |

**范围管理：** 减少范围（更少功能）以在不成比例的时间/成本增加下保持质量。

**警告：** 确实可能变慢、贵且差——避免这种结果。

## 整合检查清单

- [ ] 定义并沟通了 3 个动作聚焦支柱
- [ ] 问题陈述有具体焦点、可量化结果、清晰沟通
- [ ] 识别了核心 20% 功能用于优先开发
- [ ] 两个约束被优先，第三个被明确管理
- [ ] 范围调整以匹配可用资源
