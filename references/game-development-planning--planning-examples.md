# 详细规划示例

## 游戏支柱：创新评估

### 支柱风险评估

在定义支柱时，评估原创性与风险：

- **所有支柱与另一个游戏完全相同** → 玩家识别模式 → 被视为克隆
- **扩展、改进或独特混合支柱** → 原创性而无过度风险

### 支柱级联效应

每个支柱影响多个开发领域：

**示例："可访问性"作为支柱**
- UI 设计：清晰、大号文字和按钮
- 谜题设计：无试错机制（会违反可访问性）
- 难度：多个难度等级或自适应难度
- 控制：可重映射控制，简单输入方案

如果你在声称可访问性作为支柱的同时设计了试错谜题，你会让喜欢其他可访问元素的玩家失望。

## 问题陈述：渐进具体性

### 级别 1（太宽）
"我如何为公众设计回合制策略游戏？"

**问题：**
- 没有具体受众定义
- 没有平台约束
- 没有范围边界
- 结果将过于笼统

### 级别 2（更好）
"我如何为平板设计一款具有 F2P 商业模式的 8 位复古横版格斗游戏？"

**改进：**
- 视觉风格定义（8 位复古）
- 类型定义（横版格斗）
- 平台定义（平板）
- 商业模式定义（F2P）

### 级别 3（非常具体）
"我如何构建高级敌人巡逻 AI？"

**考量：**
- 适合具体技术问题
- 可能排除非常规解决方案
- 最好在已建立的项目背景下使用

## 80/20 法则：常见陷阱

### 陷阱 1：与帕累托最优性混淆
- 80/20 法则（原理 30）：资源分配启发式
- 帕累托最优性（原理 18）：关于有效结果的游戏理论概念
- 这些是**不同**的原理

### 陷阱 2：将 80/20 视为精确数学
- 该比率是启发式，而非精确测量
- 实际分布可能是 70/30 或 90/10
- 重要的是洞见：少数功能驱动多数参与

### 陷阱 3：忽视"不重要"的 80%
- 非核心功能仍然需要工作
- 首先打磨 20%，然后确保 80% 不会破坏体验
- 有 Bug 的菜单系统即使最好的核心游戏玩法也会毁掉

## 项目约束：高级场景

### 场景 1：固定截止日期（需要快）
**给定：** 发行商截止日期不可移动
**权衡：** 必须牺牲便宜或好
- 牺牲便宜：雇用承包商，支付加班费
- 牺牲好：削减功能，减少范围

### 场景 2：有限预算（需要便宜）
**给定：** 独立团队资金有限
**权衡：** 必须牺牲快或好
- 牺牲快：延长开发时间线
- 牺牲好：接受较低的制作价值，专注于核心机制

### 场景 3：质量要求（需要好）
**给定：** 建立声誉，旗舰标题
**权衡：** 必须牺牲快或便宜
- 牺牲快：长开发周期
- 牺牲便宜：在人才和工具上大量投资

### 神话人物月原理

来自 Fred Brooks："向延迟的软件项目增加人力会使其更延迟。"

**含义：** 某些约束无法通过投入资源解决。
- 9 个女人不能在 1 个月内生孩子
- 某些任务有固有的最短时间线
- 沟通开销随团队规模增加
