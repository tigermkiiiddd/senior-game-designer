## 第一步：穷举系统级变体

> 三步走，不能跳步。**每个变体必须回答四个问题。**

| 变体 | 描述 | 极致体验 | 最小体验 | 独特乐趣 | 乐趣重叠 |
|------|------|---------|---------|---------|---------|

| B | | | | | |
| C | | | | | |

拿到一个系统，先不要想子系统是什么。问：这个系统可以有哪些不同的玩法形态？
- 对每个变体问：做到极致是什么体验？做到最小是什么体验？
- **对每个变体问：玩家能学到什么新东西？** — 答不上来 = 这个变体缺乏乐趣基础
- **对每个变体标注独特乐趣** — 这个变体创造了其他变体无法创造的什么体验？
- 节点必须是动词（设置/选择/组合/分配），不是名词

**⚠ 常见错误 1**：只穷举「交互形式」的变体（卡片式/列表式/推荐式），而不穷举「子系统功能」的变体。交互形式是 UI 层面的，不是系统层面的。广度探索要在**系统层面**展开。

**⚠ 常见错误 2**：每个变体只写"极致/最小"，不分析乐趣来源。没有乐趣分析的变体穷举 = 功能列表，不是设计探索。**"极致/最小"描述的是规模，"独特乐趣"描述的是品质。规模不等于品质。**

|--------|------------|---------|------|

### 2.3 乐趣保护检查（不可跳过）

| 被合并的变体组 | 变体间乐趣是否不同？ | 处理方式 | 乐趣遗漏风险 |
|--------------|-------------------|---------|------------|
| A+B→子系统X | 不同/相同 | 不同→在子系统X中标注需同时容纳两种乐趣；相同→可安全合并 | 是否有乐趣在子系统定义中无体现？ |

**检查规则：**
- 功能相似 AND 乐趣重叠 → 安全合并
- 功能相似 BUT 乐趣不同 → 合并为同一子系统，但标注"需同时容纳X乐趣和Y乐趣"，在第三步中为每种乐趣设计独立的变体
- **乐趣清单中任何一项在归纳后找不到落脚点 → 标注"乐趣遗漏风险"，必须在第三步中补回**

## 模板各节说明

### 三步走规范
**顺序不能跳**：系统级变体 → 归纳子系统 → 子系统穷举。

1. **穷举系统级变体**：先穷举玩法形态，必须标注每个变体的独特乐趣。没有乐趣标注的变体穷举不合格。
2. **归纳子系统**：关键是**乐趣保护**——先列乐趣清单，再按功能归类，最后检查合并是否丢失乐趣。不能只看功能表面相似就合并。
3. **子系统穷举变体**：对每个子系统再次穷举实现方式，回溯检查第二步中的乐趣遗漏是否被补回。

### 乐趣保护的核心理念
**两个变体可能功能相似但乐趣完全不同。** 比如：
- 变体A："在地图上探索未知区域" → 乐趣是**发现未知**
- 变体B："在地图上选择不同路线到达目的地" → 乐趣是**策略规划**

两者都涉及"地图移动"，但乐趣一个是发现驱动，一个是博弈驱动。如果简单合并为"地图系统"，可能只保留了其中一个乐趣，另一个被丢弃。

**乐趣保护 = 确保归纳过程中，每种独特的乐趣都有明确的落脚点。**

### 行业案例的用法
不是功能对比表，而是**需求匹配度分析**。核心问题是：竞品解决的是哪类用户的什么需求？我们的用户有同样的需求吗？

### 边界测试
挑战的边界在哪里？和"极致/最小体验"配套使用。

### 反方案
不做这个功能，有没有替代解法？防止路径依赖。

### 利益相关方
影响谁：谁受益、谁受损？包括核心循环本身（玩家行为会不会被这个设计扭曲？）。
